# 反爬机制识别与应对详解

## 一、快速诊断流程图

```
数据采集失败
    │
    ├─ 403 Forbidden
    │   └─ IP被封/WAF硬封 → 换IP、降频、换数据源
    │
    ├─ 200但数据不变（软封）
    │   └─ WAF识别自动化 → 识别假数据、人机协作
    │
    ├─ 触发验证码
    │   └─ 行为检测 → 人机协作、打码平台
    │
    ├─ JS执行后才能看到数据
    │   └─ 动态渲染/JS加密 → bsk evaluate
    │
    ├─ 翻页后内容重复
    │   └─ 被软封 → 人机协作
    │
    └─ 都不行
        └─ WAF全面封锁 → 换数据源、豆包探测
```

## 二、五种反爬机制对比

### 1. 基础防护（⭐ 容易绕过）

**识别特征**：
- 返回403/429（请求过多）
- 需要登录才能访问
- 缺少User-Agent被拒绝

**应对方案**：
```bash
# IP限流 → 降低频率
sleep 2  # 每次请求间隔2秒

# UA检测 → 随机UA
UA="Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36"

# 请求头检测 → 补全
curl -H "Referer: https://www.example.com/" \
     -H "Cookie: key=value" \
     https://www.example.com/
```

### 2. 浏览器指纹检测（⭐⭐ 中等难度）

**检测点详解**：

| 检测点 | 原理 | bsk能否绕过 | 替代方案 |
|--------|------|-------------|----------|
| WebDriver | 检测navigator.webdriver=true | ✅ 能 | - |
| User-Agent | 检测是否真实浏览器 | ✅ 能 | - |
| Canvas指纹 | 渲染唯一指纹 | ❌ 不能 | 人机协作 |
| WebGL指纹 | 显卡渲染特征 | ❌ 不能 | 人机协作 |
| 字体列表 | 已安装字体 | ❌ 不能 | 人机协作 |
| 屏幕分辨率 | window.screen | ✅ 能 | - |
| 时区 | Date.getTimezoneOffset | ✅ 能 | - |
| 语言 | navigator.language | ✅ 能 | - |

**快速判断是否有Canvas指纹检测**：
```js
// 在bsk evaluate中执行
const canvas = document.createElement('canvas');
const ctx = canvas.getContext('2d');
ctx.textBaseline = 'top';
ctx.font = '14px Arial';
ctx.fillText('Hello', 2, 2);
console.log(canvas.toDataURL());  // 如果网站检测这个，就是Canvas指纹
```

**预警策略**：
```
如果遇到Canvas/WebGL指纹检测网站
→ 提前告知用户："这个网站可能绕不过，建议换数据源"
→ 不要浪费时间尝试绕过
```

### 3. JS混淆/加密（⭐⭐⭐ 较难）

**常见混淆类型**：

#### 3.1 变量名混淆
```js
// 原始代码
const userName = 'zhangsan';
console.log(userName);

// 混淆后
const _0x1a2b = 'zhangsan';
console.log(_0x1a2b);
```

**识别方法**：查看网页源码，JS变量名是`_0x`开头

**应对**：不需要破解，用bsk浏览器执行即可

#### 3.2 控制流扁平化
```js
// 原始代码
function test() {
  console.log('a');
  console.log('b');
}

// 混淆后
function test() {
  let _0x3c4d = 0;
  while (_0x3c4d < 2) {
    switch (_0x3c4d) {
      case 0: console.log('a'); _0x3c4d++; break;
      case 1: console.log('b'); _0x3c4d++; break;
    }
  }
}
```

**识别方法**：查看网页源码，大量switch-case结构

**应对**：用bsk浏览器执行，让JS自己还原

#### 3.3 动态Token
```
请求参数中包含加密字段：
- sign=xxx
- token=xxx
- _t=xxx

每次请求都变化 → 动态生成
```

**识别方法**：观察请求参数，每次都不一样

**应对**：
1. 用bsk evaluate执行JS，获取动态参数
2. 或者放弃，换数据源

#### 3.4 AST解析（高级混淆）
```
JS代码中包含AST解析逻辑
运行时动态生成新代码并执行
```

**识别方法**：源码中有`eval`、`Function`、`AST.parse`等关键词

**应对**：几乎无法绕过，建议换数据源

### 4. WAF防护（⭐⭐⭐⭐ 最难绕过）

**WAF工作原理**：
- 分析请求特征（IP、UA、请求频率、行为模式）
- 识别自动化行为
- 返回403或假数据

**五种封法全列**：

| 翻页方式 | 封法 | 识别方式 | 应对 |
|---------|------|---------|------|
| `/p2` 路径参数 | 硬封403 | status=403 | 换方式 |
| `?page=2` query参数 | **软封** 200但返p1数据 | 对比shopId | 人机协作 |
| 借用户标签+点下一页 | 软封 | 同上 | 人机协作 |
| agent `bsk click @eN` 下一页 | 软封 | 同上 | 人机协作 |
| **用户手动点击下一页** | ✅ 不封 | - | 唯一稳定方案 |

**软封识别代码**（必做）：
```js
// 第1步：拿p1的shopId列表
const p1 = Array.from(document.querySelectorAll('#shop-all-list li'))
  .map(li => li.querySelector('a')?.dataset?.shopid)
  .filter(Boolean);
console.log('P1店铺ID:', p1);

// 第2步：跳到p2
location.href = 'https://www.dianping.com/search/keyword/2/10_火锅?page=2';

// 第3步：等页面加载完，再拿shopId
setTimeout(() => {
  const p2 = Array.from(document.querySelectorAll('#shop-all-list li'))
    .map(li => li.querySelector('a')?.dataset?.shopid)
    .filter(Boolean);
  console.log('P2店铺ID:', p2);
  
  // 第4步：对比
  const same = p1.every((id, i) => id === p2[i]);
  console.log('是否被软封:', same);  // true=被软封
}, 3000);
```

**正确应对**（按优先级）：
1. **人机协作**：agent `bsk evaluate`读数据 + 用户手动翻页 = 唯一稳定方案
2. **接受部分数据**：拿第1页15家 + 品牌页 = 小样本
3. **换数据源**：高德API/美团API（推荐）

### 5. 验证码（⭐⭐⭐⭐⭐ 需要人工）

**常见类型**：

| 类型 | 特征 | 应对方案 | 成本 |
|------|------|----------|------|
| 图片验证码 | 扭曲字母数字 | 手动输入/打码平台 | 低 |
| 滑块验证码 | 拖动滑块 | 手动滑动/模拟滑动 | 中 |
| 点选验证码 | 点击指定物体 | 手动点选 | 中 |
| 人机验证 | "我不是机器人" | 手动完成 | 低 |
| 短信验证 | 发送验证码到手机 | 手动输入 | 高 |

**打码平台**：
- 图鉴：http://www.ttshitu.com/
- 超级鹰：https://www.chaojiying.com/
- 费率：1元=1000次左右

**人机协作方案**：
```
1. agent用bsk打开页面
2. 遇到验证码，暂停
3. 通知用户："需要手动完成验证码"
4. 用户完成后，agent继续执行
```

## 三、实战案例：大众点评WAF全面封锁

### 问题
所有自动化翻页方式都被封：
- `/p2` → 403
- `?page=2` → 软封（返回p1数据）
- 点"下一页" → 软封

### 根因
大众点评用的openresty WAF，能识别所有自动化行为

### 解决方案
**人机协作模式**：
```
1. agent用bsk evaluate读取当前页数据
2. 通知用户："请手动点击下一页"
3. 用户点击后，agent再读取数据
4. 重复，一次会话可拿120+家
```

### 代码实现
```bash
# 1. 启动会话
SID=$(bash scripts/session_start.sh)

# 2. 导航到搜索页
bash scripts/navigate.sh "https://www.dianping.com/search/keyword/2/10_火锅" "$SID"

# 3. 读取当前页数据
HTML=$(bash scripts/get-html.sh "$SID")
# 解析HTML，提取店铺信息

# 4. 通知用户手动翻页
echo "❗ 请手动点击'下一页'按钮，完成后告诉我"

# 5. 用户完成后，再次读取
HTML=$(bash scripts/get-html.sh "$SID")
# 解析...

# 6. 重复3-5，直到采集完成
```

## 四、决策树：什么时候放弃

```
反爬难度评估：

⭐ 基础防护 → 能绕过（用本文档方案）

⭐⭐ 浏览器指纹 → 部分能绕过（Canvas/WebGL绕不过）

⭐⭐⭐ JS混淆 → 看情况（能用bsk执行就继续，否则放弃）

⭐⭐⭐⭐ WAF防护 → 人机协作 或 换数据源

⭐⭐⭐⭐⭐ 验证码+WAF → 放弃，换数据源
```

**放弃信号**：
- 尝试3种方案都失败
- 破解成本 > 数据价值
- 有合规风险

**替代方案**：
- 高德API（合规+完整）
- 豆包快速探测（5分钟出大概）
- 人工调研（小样本）

## 五、Web安全基础（做方案时规避风险）

### 常见漏洞类型

| 类型 | 原理 | 简单理解 | 对数据采集的影响 |
|------|------|----------|------------------|
| SQL注入 | 用户输入拼接进SQL | 恶意输入破坏查询 | 可能误触发WAF |
| XSS | 恶意脚本注入网页 | 页面被插入代码 | 不影响数据采集 |
| CSRF | 跨站请求伪造 | 不知情时操作 | 不影响数据采集 |
| 越权访问 | 权限验证不严 | 没登录也能看数据 | **可能影响数据源选择** |

### 数据采集合规原则

**必须遵守**：
1. ✅ 只采集公开可访问数据
2. ✅ 遵守robots.txt
3. ✅ 控制采集频率（不影响目标网站）
4. ✅ 数据仅用于内部分析

**禁止行为**：
1. ❌ 破解付费内容
2. ❌ 绕过登录验证
3. ❌ 批量注册账号
4. ❌ 数据转售

**合规声明模板**（必须加到方案里）：
```
【数据采集合规声明】
数据来源：XXX网站（公开页面）
采集方式：浏览器自动化（bsk）
采集频率：每页间隔2秒
数据用途：市场分析/竞品调研（内部使用）
合规措施：
1. 遵守目标网站robots.txt
2. 控制采集频率，不影响网站正常运营
3. 不采集付费内容和非公开数据
4. 数据仅用于内部分析，不转售
```

## 六、快速参考表

### 反爬场景判断清单

| 症状 | 可能原因 | 推荐方案 |
|------|----------|----------|
| 返回403 | IP被封 | 换IP、降频 |
| 返回200但数据不变 | 软封 | 识别假数据、换方式 |
| 触发验证码 | 行为检测 | 人机协作 |
| JS执行后才能看到 | 动态渲染 | bsk evaluate |
| 翻页全被封 | WAF全面封锁 | 人机协作 |
| Canvas指纹检测 | 高级反爬 | 换数据源 |

### bsk浏览器适用场景

| 场景 | 是否适用 | 理由 |
|------|----------|------|
| 需要执行JS | ✅ | bsk能执行JS |
| 需要登录态 | ✅ | 用用户Chrome |
| 基础反爬 | ✅ | 能绕过 |
| Canvas指纹 | ❌ | 绕不过 |
| WAF全面封锁 | ⚠️ | 只能人机协作 |

### 数据获取优先级（复习）

```
1. 官方API（合规+完整+快）
   ↓ 没有API
2. 浏览器自动化（需要登录态）
   ↓ 被封
3. 公开数据整合（多源交叉验证）
   ↓ 不够
4. 人工调研（小样本）
```

---

**核心认知**：
- 不是所有数据都能拿到，知道边界很重要
- 慢一点没关系，真实数据优先
- 人机协作是终极解决方案
- 合规第一，技术第二
